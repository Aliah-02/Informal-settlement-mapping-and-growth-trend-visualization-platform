Theory

Why template inheritance?

  • Reduce duplication — layout written once in a base template
  • Consistency — navbar, sidebar, footer stay identical
  • Faster changes — update the base or an include, every page updates
  • Clear responsibility — pages only fill their unique content blocks

Theory

Base templates, blocks, extending

{# base.html #}
{% block title %}…{% endblock %}
{% block content %}{% endblock %}

{# students.html #}
{% extends "base.html" %}
{% block content %}
  …page-only markup…
{% endblock %}

extends must be the first tag. Children override blocks; they do not rebuild the whole page.

Architecture

Production-style folder layout

student-management-portal/
  config/          settings, root urls
  campus/          views, urls, sample data
  templates/
    base.html
    includes/      navbar, sidebar, header, footer, cards
    campus/        pages that extend base
  static/css|js|images
  docs/presentation.html
  TEACHING_NOTES.md
  README.md

Architecture

How a page is assembled

Shared shell

  • Sidebar + navbar + header + footer via include
  • CSS/JS loaded once in base.html
  • active_nav from context processor

Page body

  • View builds context (lists/dicts)
  • Template fills block content
  • Loops/filters render tables & cards

Implementation

Portal pages

  • Dashboard — stats cards, activity, previews
  • Students / Courses / Lecturers / Departments — dynamic tables

How to run the portal

cd student-management-portal
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver

Open http://127.0.0.1:8000/ · Slides: open docs/presentation.html in a browser

Wrap-up

Takeaways

  • Extend for page layout · Include for reusable pieces
  • Blocks keep child templates small and focused
  • Tags reverse URLs and load static assets safely
  • Filters transform data at render time
  • Clear folder organization scales to production
